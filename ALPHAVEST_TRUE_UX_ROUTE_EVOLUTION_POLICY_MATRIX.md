## 8. Route Evolution Decision Rules

| Rule ID | Rule | Effect |
| --- | --- | --- |
| RE-RULE-001 | Route count preservation is not a UX goal. | Do not reject route split/create merely because route count grows. |
| RE-RULE-002 | New route creation is allowed if page job, journey and safety are explicit. | Route evolution may appear in Flow Refactoring Plan. |
| RE-RULE-003 | P1/Hold are no longer automatic blockers. | Former P1/Hold routes enter MVP candidate set. |
| RE-RULE-004 | P1/Hold still require baseline recheck and safety recheck. | No task without current route/component/test confirmation. |
| RE-RULE-005 | A route split requires a route evolution record. | No informal split tasks. |
| RE-RULE-006 | A route merge requires no-loss proof. | Merged routes must preserve page jobs, safety context and back/recovery. |
| RE-RULE-007 | A new client projection requires no-leakage proof. | Client/internal split must be fail-closed. |
| RE-RULE-008 | A new decision room requires safety and audit proof. | Approval/release/export/committee decisions need gate and audit tests. |
| RE-RULE-009 | Deprecating from navigation is allowed if route remains accessible where needed. | Reference/P1/legacy routes can leave main nav without deletion. |
| RE-RULE-010 | Removing route is rare and requires strong proof. | Prefer deprecate/redirect over deletion unless clearly obsolete. |

## 9. Screen Split Alternative Assessment Rules

Every split candidate must compare at least `NO_SPLIT_REFINE_IN_PLACE`, `SIMPLE_SPLIT`, `MULTIPLE_SPLIT`, and `MULTIPLE_SPLIT_PLUS_MERGE`. Drawer options are added when secondary context or route/drawer conversion is plausible.

| Criterion | Weight | NO_SPLIT | SIMPLE_SPLIT | MULTIPLE_SPLIT | MULTIPLE_PLUS_MERGE | Notes |
| --- | --- | --- | --- | --- | --- | --- |
| Page Job Clarity | 5 | Kill if unclear | Strong if two jobs | Strong if 3+ jobs | Strong if duplicated jobs exist | One page job per surface. |
| Journey Continuity | 5 | Risk if user loses next step | Good if path is obvious | Needs strong in-flow nav | Good if it removes redundant jumps | Route count may grow but journey must feel simpler. |
| Cognitive Load Reduction | 5 | Only if clutter can be removed | Good for overview/detail split | Good for distinct mental modes | Good for duplicate-pattern removal | Avoid cards everywhere. |
| Navigation Simplicity | 4 | Simple but may hide structure | Often best | Needs hub/breadcrumbs | Needs redirect/nav plan | More routes must not mean more confusion. |
| Safety / Visibility Preservation | 5 | May mix client/internal context | Useful for client/internal separation | Useful for decision rooms | Allowed only with no-loss safety proof | Safety conflict overrides auto-approval. |
| Implementation Complexity | 3 | Low | Medium | High | Highest | Complexity is allowed if UX value and tests justify it. |
| Testability | 4 | Easy but may miss flow | Good | Requires route-smoke and flow tests | Requires regression and redirect tests | No tests, no task pack. |
| Backward Compatibility | 3 | Best | Needs redirect/back link | Needs route map | Needs migration/redirect plan | Existing deep links should not silently break. |
| Reuse / Component Quality | 3 | Risk of large component | Supports extraction | Supports page-type system | Supports pattern consolidation | Prefer reusable hub/workbench/detail primitives. |
| User Recovery / Back/Cancel | 4 | Can be easy | Needs clear return | Needs breadcrumbs and state preservation | Needs strong recovery map | Back, cancel, save draft and recovery remain mandatory. |

Decision rules: if `NO_SPLIT` cannot clarify page job, kill it. If `SIMPLE_SPLIT` solves the page job, prefer it. If several page jobs remain, assess `MULTIPLE_SPLIT`. If adjacent routes duplicate the same work or fragment one journey, assess `MULTIPLE_SPLIT_PLUS_MERGE`. Kill drawer-only options when safety context or decision accountability would be hidden.

| Guardrail | Applies To | Route Evolution Risk | Required Protection | Required Tests |
| --- | --- | --- | --- | --- |
| AI Draft remains internal-only | Advisory, workbench, export, client projection | New client routes accidentally expose draft/rationale | Client payload projection excludes AI/internal draft fields | no-client-leakage, payload redaction negative |
| Client sees only released/redacted/client-safe | Portal, mobile, client projection, export | Internal route split reused as client page | Separate client projection with fail-closed visibility | client visibility fail-closed |
| Advisor Approval is not Compliance Release | Advisor approval, decisions, compliance | Decision room CTA overclaims release | Separate advisor-approved and compliance-release states | workflow gate negative |
| Upload Success is not Evidence Sufficiency | Documents/evidence/upload | Upload route navigates directly to release/export | Upload-only success message and evidence review gate | document-upload + evidence sufficiency negative |
| Export Preview is not Export Approval | Export/redaction/download | Preview route doubles as approval/download | Explicit scope/redaction/preview/approval/download separation | file-export realism + forbidden payload negative |
| Admin cannot bypass gates | Admin, governance, roles, compliance, export | Governance hub exposes release/visibility mutations | Permission engine and non-bypass negative tests | permission-engine, admin-non-bypass |
| Route Access is not Payload Visibility | All routes with sensitive data | New route grants broader object fields | Object/field visibility checks and redaction rules | RBAC payload visibility negative |
| Safety conflict overrides Product Owner auto-approval | All route evolution | Auto-approval misread as safety waiver | Mark `SAFETY_REVIEW_REQUIRED` | Task pack cannot proceed without safety acceptance |

## 18. Stop Rules

- Stop if a proposed route exposes internal advice, AI draft, compliance notes, unreleased evidence or unredacted export content to client-facing users.
- Stop if route evolution weakens RBAC, object scoping, payload visibility, audit or evidence sufficiency rules.
- Stop if a split candidate has no alternative assessment.
- Stop if a new route lacks a Route Evolution Record.
- Stop if current baseline cannot be verified before a task pack.
- Stop if a Former P1/Hold route is treated as automatically safe merely because MVP elevation was approved.
- Stop if Codex would need to decide product policy, financial advice policy, compliance policy or client visibility policy.
- Do not stop merely because a route must be created, a screen must be split, a route must become a drawer, a drawer must become a route, or adjacent routes must be merged. These are allowed when records, safety and tests exist.

## 19. Decision Summary for the User

Die alte Route Policy ist nicht mehr der harte Käfig. Sie bleibt nur historisch relevant und wird überschrieben, sobald sie echtes UX-Flow-Refactoring blockiert. P1- und Hold-Routen dürfen jetzt in den MVP-Refactoring-Raum aufgenommen werden, aber jede einzelne Route braucht vorher Moving-Baseline-Check, Safety-Recheck und einen Route Evolution Record. Neue Routen, Subroutes, Hubs, Detailseiten, Decision Rooms, Client Projections, Route Splits, Route Merges und Drawer-Promotions sind erlaubt. Was nicht erlaubt ist: Safety kaputtmachen, Client-Leakage erzeugen, Advice-Gates umgehen oder ohne Tests in Codex-Tasks springen. Das nächste Artefakt ist deshalb nicht Codex, sondern `ALPHAVEST_TRUE_UX_FLOW_REFACTORING_PLAN.md`, wo die hier vorgeschlagenen Route- und Screen-Evolutionen pro Flow final entschieden werden.
