## 5. Navigation Model Decision

| Option | Strength | Weakness | Fit for AlphaVest | Decision |
| --- | --- | --- | --- | --- |
| Journey-based | Best for guiding users through evidence → review → approval → release → client visibility. | Can hide role/object context if used alone. | Very high for MVP proof path. | KEEP AS PRIMARY |
| Role-based | Matches AlphaVest actors: analyst, advisor, compliance, admin, client. | Pure role nav fragments the same journey across silos. | High as filtering and landing logic. | KEEP AS SECONDARY |
| Object-based | Good for Family Office objects: client, entity, document, recommendation, decision, evidence, export. | Object nav alone weakens process guidance. | High inside workspaces/details. | KEEP AS CONTEXT |
| Hybrid | Combines journey guidance, role-aware access and object context. | Needs disciplined hierarchy and labels. | Best fit. | KEEP / TARGET ARCHITECTURE |

**Binding navigation decision:** `HYBRID_JOURNEY_FIRST_ROLE_AWARE_OBJECT_CONTEXT`.

| Navigation Layer | Role in Target UX | Concrete Decision |
| --- | --- | --- |
| Primary mental model | Journey-first operating model. | Use workspace labels like Setup, Client Workspace, Evidence, Advisory Workbench, Approvals, Compliance, Decisions, Governance, Export. Do not expose all 71 routes equally. |
| Sidebar / App Shell | Stable workspace navigation. | Sidebar becomes grouped journeys/workspaces with badges for pending work, not a raw route list. Reference/P1/Hold are outside primary MVP nav. |
| Topbar | Tenant, role, session and search context. | Show active tenant, active role, actor type and environment; never replace journey navigation. |
| Page Header | Page job, status, primary next step. | Every MVP/MVP_SUPPORT route gets page-job text, active state, gate status, and one primary CTA. |
| Bereichs-Hubs | Orient and prioritize work. | Use hubs for Tenant Directory, Client Dashboard, Consultant Workbench, Decision List, Export start. Hubs must summarize next work and route to workbenches/details. |
| In-Flow Navigation | Step-to-step movement through a journey. | Use explicit rails/steppers for evidence lifecycle, advisory review, compliance release and export lifecycle. |
| Breadcrumbs | Object and journey context. | Use “Client → Workstream → Object → State”, not generic route breadcrumb. |
| Status/Gate Hints | Explain why action is allowed/blocked. | Status is shown before action; blocked CTAs explain missing evidence/approval/redaction/audit. |
| Client/Tenant/Object Context | Prevent context loss and leakage. | Context chips live in topbar/header; object detail pages show object scope and visibility caveat. |

## 17. Decision Summary for the User

Die Entscheidung ist klar: AlphaVest sollte ein **Hybrid-Navigationsmodell** bekommen — aber nicht beliebig hybrid, sondern **journey-first als Hauptführung**, **rollenbewusst in Landing, Sichtbarkeit und Aktionen**, und **objektbezogen innerhalb der Arbeitsseiten und Details**.

Das Page-Modell ist ebenfalls entschieden: **Hubs orientieren**, **Workbenches bearbeiten**, **Detailseiten entscheiden**, **Drawers liefern Kontext**, **Modals bestätigen sicherheitskritische Aktionen**, **Referenzseiten bleiben Referenzseiten**, **P1 bleibt deferred**, und **Hold bleibt blockiert**.

Das Density-Zielbild ist **Controlled Premium Workbench Hybrid**: Client- und Executive-Oberflächen bleiben ruhig und hochwertig; interne Analyst-/Advisor-/Compliance-/Governance-Flows werden deutlich kompakter, tabellarischer und handlungsorientierter. Premium darf nicht leer wirken, Workbench darf nicht chaotisch wirken.

Die CTA-Logik folgt einem festen Prinzip: **genau ein primärer nächster Schritt pro State**. Upload heißt nur Upload, nicht Evidence Sufficiency. Advisor Approval heißt nicht Release. Compliance Release heißt nicht Client Acceptance. Export Preview heißt nicht Approval oder Download. Blockierte CTAs müssen erklären, was fehlt und wo der Nutzer es behebt.

Die neue ZIP-Evidence mit 568 Screenshots reicht stark aus, um Page-Type-, Density- und CTA-Entscheidungen zu treffen. Von den zwei gleichnamigen Videos ist technisch nur eine Datei eindeutig greifbar; ich habe sie als `VIDEO_A_JOURNEY_PRIMARY` benannt. Das zweite, inhaltlich andere Video bleibt als `VIDEO_B_JOURNEY_SECONDARY` geführt und sollte bei Bedarf unter eindeutigem Dateinamen nachgereicht werden. Die Entscheidung wird dadurch nicht blockiert, weil ZIP + Video A + bestehende AlphaVest-Artefakte für diese Ebene ausreichend sind.

Nächster sinnvoller Schritt: `ALPHAVEST_UX_REFACTORING_CODEX_TASK_MASTER.md` erzeugen — mit Task-Gruppen für Navigation/App Shell, Hubs, Page Complexity, CTA/Next-Step, State/Feedback und Density-System. Erst danach sollte ein konkreter UX-Codex-Implementation-Handoff entstehen.
