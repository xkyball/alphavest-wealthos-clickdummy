## 1. Executive Decision

**Artefact status:** `FEEDBACK_VALIDATION_ERROR_STATE_CONTRACT_ACCEPTED_WITH_DOWNSTREAM_DEPENDENCIES`

**Codex readiness status:** `CODEX_HANDOFF_NOT_READY`

This artefact specifies feedback, validation, loading, success, error, permission, empty, blocked, retry, upload, release, evidence, audit, client-visibility and export feedback behaviour for AlphaVest WealthOS. It converts the locked route scope, route/screen/visual matrix, missing screen/state-screen decision log, state-screen specification and drawer/modal interaction contract into a concrete feedback contract without implementing anything.

The central rule is strict: feedback must never overclaim product readiness. Upload success is not evidence sufficiency. Advisor approval is not compliance release. Compliance release is not client acceptance. A visible audit timeline is not audit persistence proof. A button is not a mutation. A status chip is not a gate. A demo action is not production persistence.

This artefact does **not** create code, tests, APIs, Prisma changes, migrations, routes, screens, state-screen images, visual assets, Codex tasks or a final Codex handoff.

## 5. Feedback / Validation Contract Method

1. Preserve roadmap order and keep Codex blocked.
2. Use locked route worksets; do not reclassify route scope.
3. Translate state-screen requirements into feedback semantics.
4. Translate drawer/modal/upload/action lifecycles into validation, loading, success, error and retry behaviour.
5. Fail closed for safety-sensitive states: client visibility, advice boundary, RBAC, evidence, audit and export.
6. Route unresolved safety, API, schema and test proof to later artefacts without implementing.
7. Treat feedback wording as a safety surface: no success message may imply a gate that has not actually passed.

## 7. Cross-Cutting Feedback Rules

| Rule ID | Rule | Required Feedback Behaviour |
|---|---|---|
| FVE-RULE-001 | No success overclaim | Every success message names only the completed action and explicitly avoids implying downstream safety gates. |
| FVE-RULE-002 | Fail-closed client visibility | When release/redaction proof is incomplete, client-facing routes show hidden, redacted, denied or empty state, not internal data. |
| FVE-RULE-003 | Upload is upload only | Upload success confirms file transfer only; evidence sufficiency, review and release remain locked. |
| FVE-RULE-004 | Advisor approval is not release | Advisor success feedback must not imply compliance clearance or client visibility. |
| FVE-RULE-005 | Compliance release is not client acceptance | Release success must not imply the client accepted, understood or acted on a decision. |
| FVE-RULE-006 | Audit display is not persistence proof | Audit timeline visibility must not be presented as immutable persistence unless later audit contract/test proof exists. |
| FVE-RULE-007 | Status chips are not gates | Status chips may label state but cannot be the proof of gate enforcement. |
| FVE-RULE-008 | Buttons are not mutations | A clickable or visible button is not treated as production mutation unless API/schema/test proof exists later. |
| FVE-RULE-009 | Demo actions are labelled as limited | Partial typed-workflow actions must have feedback that avoids production persistence claims. |
| FVE-RULE-010 | Hold routes remain blocked | Routes 064–067 and 069–071 receive no finalized MVP feedback contract. |
