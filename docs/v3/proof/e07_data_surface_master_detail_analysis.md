## Current Pattern Reality

E07 is not a greenfield component problem. The repo already has partial shared primitives, but adoption is uneven:

- `components/ui/data-table.tsx` is the main shared data-table primitive. It supports compact rows, mobile cards, sortable columns, row actions, disabled row-action messaging and non-ready state projection through `StatePanel`.
- `components/ui/filter-bar.tsx` provides a reusable search/filter shell, but many live surfaces still define local search and static filter controls.
- `components/ui/kanban.tsx` exists as a shared board primitive, but key board-to-detail flows still render local board cards and local detail drawers.
- `components/worksurface-shell.tsx`, `components/ux-detail-standard-panel.tsx` and `lib/ux-density.ts` already define density, sticky rail and detail placement concepts.
- E01-E06 have already produced a stronger contract stack for operating model, page templates, review mode, state lifecycle, action hierarchy and feedback messaging. E07 should project from that stack instead of adding another visual-only layer.

## Findings

1. The shared `DataTable` is widely adopted across internal workflow, suitability, tenant setup, committee review, monitoring, communication/export ops, client intake, decisions governance and KYC surfaces.
2. Shared adoption is incomplete. Manual `<table>` implementations still exist in high-impact governance/admin surfaces, including advice boundary, permission matrix and role matrix patterns. These local copies use table styles and row-action metadata that should be generated by the shared primitive or a deliberate specialized variant.
3. Row action hierarchy is mostly present but under-specified. `DataTable` always reserves an actions column and marks disabled row actions, but it does not yet distinguish open-detail, preview, command, blocked command and no-action row policies as a first-class contract.
4. Density exists as route/page policy in `lib/ux-density.ts`, but table/list components do not consume named data-surface density presets. Current usage still relies on boolean `compact` and route-local class choices.
5. Filter/search state is fragmented. `FilterBar` has canonical inactive static-filter messaging, while route-level queues implement their own search inputs, disabled filter buttons, clear buttons and active-state behavior.
6. Master-detail behavior exists as route-local layout, not as a canonical pattern. Advisor/compliance queues, wealth actions and committee review use selected objects, sticky rails, drawers or detail sections, but the selection-to-detail contract is not reusable or inspectable.
7. Sticky rail conventions are present in worksurface/detail components, yet route-local asides still repeat sticky class stacks. This makes future screenshot/capture validation depend on local discipline instead of a shared contract.
8. Board/list-to-detail flows are especially exposed to drift. The wealth action board uses local board columns, cards, filters, grouping and drawer behavior even though `Kanban`, `WorksurfaceShell`, E04 lifecycle states and E05 action hierarchy already provide reusable building blocks.
9. Empty/loading/error/permission states are handled cleanly when `DataTable` state is used, but manual tables and local board/list implementations can bypass the state taxonomy.
10. The strongest cleanup path is to make `DataTable`, `FilterBar`, `Kanban` and master-detail shell conventions canonical, then retire route-local table/filter/board copies unless a documented exception is approved.

## Systemic Pattern Families

- Table/list density: compact operations tables, standard review queues, detail-support tables, mobile card projection and row-height guarantees.
- Column priority: identity column, state/status column, due/risk/evidence columns, secondary metadata, hidden mobile columns and action column.
- Row actions: open detail, preview, command handoff, blocked static control, disabled unavailable control and no row action.
- Filter/search: search value, active query, active filter chips, disabled static filters, clear/reset action and filter state persistence boundary.
- Master-detail: selected row/card, detail rail, drawer/detail route handoff, selected-object summary, evidence/audit side content and blocked action boundary.
- Board/list flows: board columns, card selection, column actions, grouping, state counts and detail drawer/rail.
- States: ready, empty, loading, error, denied/restricted, hidden/internal-only, blocked and audit/export unavailable.
- Sticky headers/rails: top navigation, table header, action rail and detail rail placement rules.

## Recommended Specification Need

The spec should define one canonical data-surface contract, not a screen-by-screen redesign list.

The contract should answer:

1. Which data-surface family is being rendered: table, list, board, queue, audit log, redaction table, permission matrix or detail-support list?
2. Which density preset applies and what row/card/header dimensions are allowed?
3. Which columns are priority, secondary, mobile-hidden or action-only?
4. Which row action policy applies and which downstream claim is forbidden?
5. Which search/filter state is active, inactive, disabled-static or resettable?
6. Which master-detail mode is allowed: inline rail, drawer, route handoff or no detail?
7. Which empty/loading/error/permission state contract must render?
8. Which sticky header/rail behavior is allowed for the surface?
9. Which route-local table/filter/board implementations must be migrated or documented as exceptions?

## Risks

- A route-by-route table cleanup would preserve the main debt: every large screen would keep inventing its own density, filters and action semantics.
- A new visual wrapper without retiring local copies would make the component stack heavier while leaving the old vocabulary alive.
- Over-aggressive migration before the spec could accidentally change route scope, action authority or client/internal visibility.
- Master-detail refactoring can create hidden safety risk if a selected row appears to authorize release, export, advice or client visibility.
- Sticky headers/rails can regress mobile usability if dimensions and overflow are not specified as part of the contract.

## Open Decisions For Specification

- Whether to approve a new canonical `lib/ux-data-surface-contract.ts` as the typed source of truth for data surfaces, density, column priority, row actions, filter state and master-detail modes.
- Whether route-local manual tables should be treated as temporary exceptions that must reference the canonical contract, or as deprecated implementations to be migrated immediately when touched.
- Whether `components/ui/data-table.tsx` should absorb all table variants, or whether a small companion `DataSurfaceShell` should coordinate table/list/board/filter/detail metadata.
- Which representative surfaces should be migrated first after approval: recommended first slice is advisor/compliance queues plus one governance/admin manual matrix and the wealth action board.

## Acceptance Inputs For `SPEC-E07-1`

- E07 should build on E01 operating model, E02 page template, E03 review/proof separation, E04 state lifecycle, E05 action hierarchy and E06 feedback messaging.
- E07 should not add routes, backend pagination, new API behavior, schema changes, permission-engine behavior or product-scope decisions.
- Shared primitives should expose runtime attributes for density preset, surface family, row-action policy, filter state and master-detail mode.
- Manual table/filter/board copies should either migrate to shared primitives or carry a documented exception marker tied to the canonical contract.
- Row actions must not imply release, export, advice, approval, client visibility or mutation unless the action hierarchy contract authorizes that meaning.
- Empty/loading/error/permission states must project through the E04/E06 state and feedback contracts.
- QA should include source-level contract tests and representative runtime checks for table density, row actions, filter state, sticky rail/header behavior and local-copy retirement.

## Recommended Decision Direction

Approve the bold cleanup path: make a real typed data-surface contract and require shared table/list/board/filter/master-detail primitives to project from it. Do not preserve route-local manual tables and filter bars as equal vocabulary. Keep exceptions explicit, temporary and test-visible.
