Executive orientation

What AlphaVest coordinates

AlphaVest WealthOS Collaborative Workflow User Manual V3 explains AlphaVest as a collaboration system, not as a catalogue of screens. The product value is that multiple roles can move from fragmented inputs to reviewed, released, evidenced, and auditable outcomes without confusing internal progress with client-visible advice.

Digital first

Work enters the system as structured tenant, profile, entity, document, message, signal, decision, or export context.

Human reviewed

Analysts, advisors, compliance, privacy, and security each inspect the part of the workflow they are responsible for.

Evidence backed

Important actions create traceable proof so future reviewers know why the workflow advanced.

Client release controlled

Client visibility waits for the required release, permission, evidence, and sensitivity gates.

Collaboration model

How work moves through AlphaVest

Every important flow follows the same pattern: a role introduces or improves an object, the system changes state, the next role sees a controlled view, and evidence or audit explains why the relay is legitimate.

01

Input

Tenant, profile, entity, document, message, or request data

Roles
Principal, Family CFO, Client, External Advisor, Ops Admin
State movement
draft/new/empty -> submitted/uploaded/onboarding
Other roles see
Internal teams see a scoped work item, data gap, or review-ready object inside the tenant context.
Protected from view
Advice-like conclusions, unreleased recommendations, restricted documents, and unapproved exports remain hidden.
02

Normalization

Raw profile fields, entity details, extraction output, trigger context, or queue item

Roles
Analyst, Client Success, System
State movement
submitted/uploaded/ai_extracted -> analyst_review/client_confirmed/in_review
Other roles see
Reviewers see structured context instead of raw fragments: who owns it, what is missing, what is sensitive, and what action is next.
Protected from view
Client-facing advice, compliance release, and final evidence packages remain unavailable.
03

Review

Normalized signal, document, recommendation candidate, export scope, or governance request

Roles
Analyst, Advisor, Compliance, Privacy Officer
State movement
analyst_review -> advisor_review/compliance_pending/needs_evidence
Other roles see
The next role sees why the item is ready, what evidence supports it, and which gate still controls visibility.
Protected from view
Client-visible release stays blocked until compliance and permission conditions are satisfied.
04

Gate

Advice-like recommendation, sensitive access change, export package, or client-visible message

Roles
Compliance, Security, Privacy Officer
State movement
compliance_pending/approval_required -> released/blocked/needs_evidence
Other roles see
Approved material becomes available to the correct downstream role; blocked items return with a reason and next legitimate action.
Protected from view
Anything blocked, restricted, or redacted remains hidden from unauthorized roles.
05

Visibility

Released recommendation, decision, evidence record, message, or export

Roles
Compliance, Permission engine, Client-facing roles
State movement
internal_only/advisor_visible/compliance_visible -> client_visible or redacted
Other roles see
Clients see only released, scoped, or redacted material; internal teams see full lineage and control history.
Protected from view
Internal notes, unreleased drafts, raw extraction data, and restricted evidence remain internal.
06

Decision

Released decision object with evidence context

Roles
Principal, Family Council, Trustee scoped
State movement
released_to_client -> accepted/deferred/rejected/evidence_created/review_scheduled
Other roles see
Advisor, compliance, and operations see the decision outcome, evidence link, and review cadence.
Protected from view
Unrelated internal review notes and restricted evidence remain hidden.
07

Evidence

Decision, compliance release, document, approval, communication, export, or audit event

Roles
Analyst, Advisor, Compliance, Privacy Officer, System
State movement
initial evidence stub/created -> linked/validated/released/restricted
Other roles see
Each role sees proof appropriate to its permission and visibility scope.
Protected from view
Highly restricted or internal-only evidence is hidden or redacted.
08

Audit

Sensitive action, state transition, release, block, export, access change, or denied action

Roles
Security, Compliance, Ops, Product, QA
State movement
action attempted -> audit result success/denied/blocked/failed/pending/warning
Other roles see
Control owners see lineage, result, actor role, target object, previous state, next state, and reason.
Protected from view
Audit details remain internal unless released as an approved evidence or export item.
09

Monitoring

Queue item, SLA breach, data quality issue, roadmap scope, or state reference

Roles
Ops, Product, QA, Leads
State movement
open/in_review/blocked -> resolved/deferred/escalated/completed
Other roles see
Teams see where workflow friction accumulates and which service or product action should follow.
Protected from view
Reference and operational pages remain internal-only.

Role relay model

Who changes what the next role can see

These relays make collaboration explicit. They show where a workflow crosses from one role lens into another and why that handoff protects visibility, release, and accountability.

RR-001

Platform Admin -> Compliance

Platform policy baseline

Why first role acts
The platform role establishes the rule set that later tenant, evidence, export, and release workflows inherit.
What changes
Policy settings move from draft or review candidate into a reviewed baseline.
Next role sees
Compliance sees advice-boundary, evidence, release, and export rules that determine whether later work can become client-visible.
Client visibility impact
Indirect but foundational: later client visibility depends on this baseline.
RR-002

Compliance -> Security Officer

Sensitive platform controls

Why first role acts
Compliance confirms the policy meaning; security confirms access, session, and second-confirmation controls.
What changes
Sensitive configuration remains blocked until confirmation and audit expectations are satisfied.
Next role sees
Security sees which policy areas affect permission, MFA/session, external access, and export behavior.
Client visibility impact
Prevents unsafe visibility defaults from being inherited by tenants.
RR-003

Ops Admin -> Client Success

ClientTenant onboarding

Why first role acts
Ops creates the tenant shell so a family office can be configured in a scoped environment.
What changes
Tenant moves from draft to onboarding readiness.
Next role sees
Client Success sees owner assignments, missing activation gates, invitation readiness, and tenant policy profile.
Client visibility impact
No client-facing workspace is reliable until activation gates are satisfied.
RR-004

Client Success -> Invited User

Invitation and onboarding consent

Why first role acts
Client Success sends the invitation only after tenant setup has enough owner, policy, and role context.
What changes
User moves from invited toward active/demo role-confirmed state.
Next role sees
The invited user sees identity, privacy acknowledgement, consent, and role confirmation surfaces.
Client visibility impact
Creates scoped access context for later client-side visibility.
RR-005

Principal / Family CFO -> Analyst

Client profile and family context

Why first role acts
Client-side roles provide facts, relationships, preferences, and structure context that internal teams cannot safely infer.
What changes
Profile and relationship data become reviewable tenant context.
Next role sees
The analyst sees a structured family context with missing evidence, conflicts, and review needs.
Client visibility impact
Client input can remain visible to the client, but internal interpretation and advice remain gated.
RR-006

Analyst -> Advisor

Entity, document, signal, or recommendation candidate

Why first role acts
Analysts normalize data chaos into structured, reviewable context before advisory judgement is applied.
What changes
Workflow moves from analyst review toward advisor review or awaiting information.
Next role sees
Advisor sees the candidate package, source context, evidence status, and unresolved blockers.
Client visibility impact
Advisor approval is internal human review, not client visibility.
RR-007

Client / External Advisor -> Analyst

Document upload and extraction

Why first role acts
The source document enters the system from the party that has it, but extracted values must be reviewed before they become trusted context.
What changes
Document moves uploaded -> ai_extracted -> client_confirmed -> analyst_review_pending.
Next role sees
Analyst sees document type, sensitivity, extracted values, confidence, corrected values, and entity links.
Client visibility impact
Clients may see simplified document state; internal extraction detail is not automatically client-visible.
RR-008

Advisor -> Compliance Officer

Advisor-approved recommendation package

Why first role acts
Advisor review supplies human judgement and rationale, but the package still requires release control.
What changes
Recommendation moves advisor_approved -> compliance_pending.
Next role sees
Compliance sees advisor rationale, evidence completeness, classification, permissions, and release/block controls.
Client visibility impact
This relay is the decisive client-visibility gate.
RR-009

Compliance Officer -> Principal / Family Council / Trustee scoped

Released client decision

Why first role acts
Compliance releases only the version that is safe, scoped, evidenced, and permitted for client review.
What changes
Decision moves released_to_client -> awaiting_family_approval or decision-ready.
Next role sees
Client-side participants see the released decision, evidence summary, required participants, and accept/defer/reject controls.
Client visibility impact
Client visibility is now permitted but still role/participant scoped.
RR-010

Principal / Admin -> Compliance / Security

Governance access request

Why first role acts
A user or admin requests changed access because collaboration requires the right people to see the right objects.
What changes
Access request moves requested -> in_review -> approved/blocked/revoked.
Next role sees
Compliance/Security sees requester, target user, scope, sensitivity, reason, and second-confirmation need.
Client visibility impact
Access changes alter who can see client-visible, restricted, or internal objects later.
RR-011

Advisor / Client Success -> Client

Communication escalation

Why first role acts
The internal team chooses the right channel when digital-only work is no longer enough.
What changes
Communication moves draft/scheduled -> sent/completed/evidence-linked when material.
Next role sees
Client sees only the approved or appropriate message/call context, not internal notes.
Client visibility impact
Channel choice determines what is client-visible and what remains internal service context.
RR-012

Advisor / Compliance -> Privacy Officer

Export request and redaction

Why first role acts
The business wants to share evidence or decision material outside the live UI, but export creates risk.
What changes
Export request moves draft -> scope_selected -> redaction_pending -> approval_required/approved -> generated/downloaded.
Next role sees
Privacy Officer sees export type, scope, redaction profile, external sharing context, expiry, and approval state.
Client visibility impact
Exports create a controlled external visibility surface separate from the app UI.
RR-013

Ops -> Product / QA / Leads

Queue, SLA, roadmap, and state friction

Why first role acts
Ops observes bottlenecks and service friction that reveal where the workflow model needs improvement.
What changes
Queue/data quality issue moves open -> in_review/blocked/resolved/deferred.
Next role sees
Product and QA see which states, queues, or routes create friction and which gaps are implementation limits.
Client visibility impact
Mostly internal; improves future consistency and reduces client-facing confusion.

Workflow story WS-001

Tenant becomes operational

Workflow purpose

A family-office workspace can become active before ownership, compliance, policy, and invitation context are aligned.

Roles involved

Platform Admin, Ops Admin, Client Success, Compliance, Invited User

Reader takeaway

A tenant is not just a record. It is the scoped collaboration container that makes later visibility and accountability possible.

Handoff sequence

  • Platform Admin establishes baseline
  • Ops Admin creates tenant
  • Client Success coordinates readiness
  • Compliance confirms owner/policy gates
  • Invited User accepts and acknowledges scope

State changes

  • TenantStatus: draft
  • TenantStatus: onboarding
  • UserStatus: invited
  • UserStatus: active/demo-confirmed

Visibility changes

  • Internal setup visible to AlphaVest roles
  • Invited user sees only onboarding context
  • Client workspace is not treated as operational until gates are satisfied

Evidence and audit

  • Policy changes audit
  • Tenant create/team assignment audit expected
  • Invitation and consent audit expected

Required inputs and objects

PolicyDefinition, ClientTenant, User, UserRole, AuditEvent

What the system prevents

tenant activation without required owners,role context without invitation/consent,unscoped client data

Demo boundary: The UI is navigable with demo data; broad persisted tenant governance transactions are not fully claimed.

Evidence and audit lineage

How AlphaVest preserves the reason a workflow advanced

Evidence lets a later reviewer understand what changed before they entered the flow. Audit events protect sensitive actions by preserving actor, role, object, previous state, next state, and result.

Review memory

Evidence connects decisions, releases, documents, exports, messages, and approvals into a coherent trace.

Controlled correction

Corrections create new events rather than silently rewriting what happened.

Future collaboration

The next role can act with context instead of reconstructing the past from messages or memory.

Client trust

Released material has an explainable lineage instead of appearing as an unsupported conclusion.
